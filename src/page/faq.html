Any plans to collaborate with Halium?

Yes, I hope we can share udev rules, unify kernels and maybe work on a few other parts together! But keep in mind, that postmarketOS has a different architecture (mostly due to Alpine Linux, which is required for pmbootstrap), so it can not build on the Halium stack and this will not be a goal in the future. It does not use the glibc, systemd or the Android build system, or run Android next to its Linux kernel, Musl libc and busybox userland.

Why did you wait so long before releasing anything?

I needed to make sure, that pmbootstrap is safe to use. It downloads Alpine's apk (Alpine Linux package manager, not to be confused with the Android app format) binary from the Internet and executes it with root rights. This means, that apk needs to be verified properly before execution (GPG key signature check with openssl, minimum required version), and the whole verification process needed to have testcases (which it does now). Furthermore I took my time to put up this blog and write the announcement, as well as the rest of the content.

Oh and I did actually release my Alpine cross-compiling efforts about a month earlier.

Why do you keep talking about Musl/Linux all the time instead of GNU/Linux, why does Alpine use musl and BusyBox instead of glibc and the GNU Core Utilities?

GNU/Linux implies that the system is based on the Linux kernel, and then uses GNU software (the GNU LibC, and the GNU coreutils) to put that kernel to some use. An astonishing part of the software in normal GNU/Linux distributions comes from the GNU project, and no "Linux distribution" would be usable without GNU. Except Alpine, and all the other distributions based on alternative libc libraries and userlands. Indeed, Alpine has replaced a lot of GNU components (with the exception of GCC and the binutils, which are in the works). But they have also packaged just about every typical GNU userspace component, except for the glibc; so you can just install them if you want to (example: apk add bash). Having a different libc means that software compiled for GNU/Linux will not necessarily run on Alpine. Since there already is a heavy confusion between Linux and GNU/Linux, it appeared important to make that distinction clear: while you can install GNU software from Alpine packages on Alpine, you cannot take any binary linked against the GlibC (steam, skype, etc.) and simply have it work. Besides that, calling postmarketOS a Musl/Linux distribution helps to differentiate it better:
  • GNU/Linux (Linux kernel with a GNU userspace)
  • Android/Linux (Linux kernel with Android Bionic libc, and Android userspace)
  • Musl/Linux (Linux kernel with the Musl libc, useless without a userspace such as busybox)
  • Linux (just the kernel)
